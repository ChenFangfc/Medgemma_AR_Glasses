Return = single press

Menu = long press (2s)

Wheel scroll = Left/Right

Escape = double press, but not reliable / may be OS Back, so we treat it as optional/non-critical

Also: in Scheme 1 we are not doing “single toggles short/full” and “double toggles note/advice” anymore. Instead we use a single-cycle on Return that goes through the four views:

NOTE_SHORT → ADVICE_SHORT → NOTE_FULL → ADVICE_FULL → (repeat)

And we use long press to toggle recording.

AR Glasses Clinical Assistant — Requirements Specification (v1.1)
1. Purpose

This AR glasses system assists a clinician during patient encounters by:

Capturing spoken audio from the clinician–patient conversation.

Sending audio to a server-side workflow via WebSocket.

Receiving structured, screen-friendly outputs generated by:

MedASR (speech-to-text)

MedGemma (summarization + documentation + decision-support suggestions)

Displaying results in a compact, readable format optimized for a small AR display.

The system must minimize clinician cognitive load, reduce interaction complexity, and ensure critical information is quickly accessible.

2. System Overview
2.1 Back-end Services (Server)

The server exposes WebSocket services as defined in Model_WS_Services.md.

Primary workflow entrypoint for the AR client:

Workflow Service: ws://<server-ip>:8003/ws
(server_pipeline.py: ASR → Gemma → store outputs)

For each audio turn, the server generates and stores four outputs:

note_short — compact clinical note (default AR view)

note_full — expanded clinical documentation note (structured)

advice_short — compact decision-support suggestions (on-demand)

advice_full — expanded decision-support suggestions (structured, on-demand)

The AR client is not responsible for inference; it only records audio, sends requests, and renders returned text.

2.2 AR Client (Glasses)

The AR client controls audio capture, UI state, and on-demand content requests. It does not run ASR or LLM inference locally.

3. Input Device and Interaction Model
3.1 Verified Hardware Input Capabilities (Unity-Level)

Based on device testing, the rotary wheel yields these Unity key events:

Wheel Scroll: LeftArrow / RightArrow

Wheel Single Press: Return

Wheel Long Press (~2s): Menu

Wheel Double Press: typically maps to Escape (may be handled by OS; not reliable for critical actions)

3.2 Interaction Mapping (Required, Scheme 1)
Long press (2 seconds) — Toggle Recording (Critical)

Long press toggles recording state: Start / Stop.

On Stop, the device sends the recorded audio to the workflow WebSocket.

Recording must not be triggered accidentally by scrolling or short presses.

Single press — Cycle Display Views (Primary Navigation)

Each single press cycles the view in this exact order:

NOTE_SHORT (default)

ADVICE_SHORT

NOTE_FULL

ADVICE_FULL

Repeat from (1)

Notes:

Advice is never shown unless the clinician cycles to it.

Cycling must not trigger inference; it only changes what is displayed and may request cached/missing content.

Wheel scroll — Scroll Within Current View

Scroll up/down within the currently displayed text.

Scrolling must not change view mode or trigger recording.

Double press (Escape) — Optional / Non-Critical Only

Double press may be intercepted by OS Back behavior.

If used, it must be non-critical (e.g., “return to NOTE_SHORT”) and must not stop recording or trigger upload.

4. Recording Feedback Requirements (Audio + Visual)
4.1 Audible Feedback (“Earcons”)

Requirement: each time recording is started or stopped, the glasses must play a brief audible cue.

On recording start: short “start” beep (~80–150 ms)

On recording stop: short “stop” beep (distinct from start, e.g., different pitch or double-beep)

Optional (recommended):

Volume setting (Low/Medium/High) and/or Silent Mode

4.2 Recording-In-Progress Visual Indicator

Requirement: while recording is active, the UI must show a clear recording indicator on all views.

At least one (recommended: both):

Recording timer: elapsed time in mm:ss, update ≥ 1 Hz

Pulsing/breathing icon: small REC ● or mic icon that pulses steadily

Constraints:

Indicator visible on all views (NOTE_SHORT, NOTE_FULL, ADVICE_SHORT, ADVICE_FULL)

Must not obscure key text content

On recording start: show immediately

On recording stop: hide immediately and show “Processing…” state until results arrive

4.3 Optional: Haptic Confirmation

If hardware supports haptics, provide a short haptic tick on recording start/stop.

5. UI Views and Display Requirements
5.1 Supported Views (Exactly Four)

The UI must support exactly these display states:

NOTE_SHORT (default)

NOTE_FULL

ADVICE_SHORT

ADVICE_FULL

These map directly to the server outputs.

5.2 Default Display Behavior (After Each Turn)

After each successfully processed audio turn:

Automatically display NOTE_SHORT for the latest turn, regardless of what view was previously open.

Rationale: clinicians should always see the newest compact note first.

5.3 Layout Rules for Short Views (Glasses-Friendly)

For note_short and advice_short:

Max 6 lines total

Line 1: single-sentence summary (≤ 80 characters)

Exactly 3 bullet lines beginning with • (≤ 16 words each)

Optional last line begins with ⚠ only when urgent red flags exist

5.4 Layout Rules for Full Views

For note_full and advice_full:

Must be scrollable

Must remain structured and readable (section-like formatting or key/value blocks)

Must not include model meta text:

No confidence checklists

No chain-of-thought / reasoning traces

No “mental sandbox” or internal planning text

5.5 Minimal UI Elements

The UI must display:

Recording indicator (Section 4)

Processing indicator: “Processing…” while awaiting results

View header showing current state:
NOTE (SHORT) / ADVICE (SHORT) / NOTE (FULL) / ADVICE (FULL)

Connectivity indicator (recommended): Online / Offline

6. State Management Requirements
6.1 Client State Variables

The AR client must maintain:

recording: boolean

current_view: enum { NOTE_SHORT, ADVICE_SHORT, NOTE_FULL, ADVICE_FULL }

session_id: string (issued by server)

turn_index: integer (latest turn number, if provided)

cache: in-memory cache of the four outputs for the current session/turn

6.2 View Transition Rules

Single press: advance current_view in Scheme 1 order

Scroll: scroll within current_view

New turn_result arrives: force current_view = NOTE_SHORT and display latest note_short

7. WebSocket Communication Requirements
7.1 Primary Endpoint

The AR client connects to:

ws://<server-ip>:8003/ws (or wss://<domain>/ws when tunneled via TLS)

7.2 Connect-time Banner Handling

If the server sends an informational banner on connect (e.g., op:"ready"), the client must ignore it and proceed.

7.3 Start Session Flow

Request:

{"op":"start_session","request_id":"s1","patient_id":"<optional>"}


Response:

{"op":"session_started","request_id":"s1","session_id":"...","turn_count":0}

7.4 Process Audio Flow (Default Path)

Triggered when recording stops.

Request:

{
  "op":"process_audio",
  "request_id":"t1",
  "session_id":"...",
  "audio_b64":"<base64 audio bytes>",
  "sample_rate":16000,
  "return":["note_short"]
}


Response:

{
  "op":"turn_result",
  "request_id":"t1",
  "session_id":"...",
  "turn_index":1,
  "note_short":"...",
  "available":["note_full","advice_short","advice_full"]
}


Default UI behavior: display note_short immediately (and set view to NOTE_SHORT).

7.5 On-Demand Fetch Flow (For View Cycling)

When the clinician cycles into a view that is not cached locally, request it from the server.

Request:

{"op":"get_latest","request_id":"g1","session_id":"...","what":"note_full"}


or:

{"op":"get_latest","request_id":"g2","session_id":"...","what":"advice_short"}


or:

{"op":"get_latest","request_id":"g3","session_id":"...","what":"advice_full"}


Response:

{"op":"get_result","request_id":"g2","session_id":"...","what":"advice_short","text":"..."}

7.6 Caching Behavior (Recommended)

Cache any successfully fetched content per session/turn.

Do not re-request if cached unless clinician explicitly refreshes (refresh is optional future feature).

7.7 End Session Flow (Optional)

Request:

{"op":"end_session","request_id":"e1","session_id":"...","include_transcript":true}


Response:

{"op":"session_summary","request_id":"e1","session_id":"...","session_closed":true}

8. Data Storage Requirements (Server-Side)

The server must store generated outputs locally per patient/session/turn. The AR client should not persist patient data long-term.

Recommended server structure:

/srv/local/chenf3/patients/<patient_id>/sessions/<session_id>/
  transcript.jsonl
  turns/0001.json
  turns/0002.json


Each turn file should include:

timestamp

ASR transcript (asr_text)

note_short, note_full

advice_short, advice_full

latencies and request metadata (optional)

9. Error Handling Requirements
9.1 Connection Errors

If WebSocket connect fails:

Show Offline — retry

Allow user retry

UI must not crash or freeze

9.2 Processing Errors

If server returns op:"error":

Show “Processing failed”

Optionally show details in a debug view

9.3 Timeouts

If server does not respond within configured timeout:

Show “Timeout — try again”

Allow retry (re-send audio if buffered)

9.4 Recording While Offline (Recommended)

If user starts recording while offline:

Play start earcon + show recording indicator

Record locally

On stop, attempt upload; if upload fails show “Upload failed — retry”

10. Performance and UX Requirements

Optimize default path (Stop recording → display note_short) for low latency.

Avoid requesting full outputs unless explicitly needed (via view cycling).

UI must remain responsive during audio upload and server processing.

Clinician should be able to:

start/stop recording reliably (long press)

see the short note immediately after processing

access advice with minimal clicks (single press cycle)

11. Privacy and Security (Baseline)

Prefer patient_id over real names for server storage directories.

AR client should not permanently store patient data locally.

Restrict WebSocket access to trusted networks; TLS/auth can be added later.

12. Acceptance Criteria (Definition of Done)

The system is successful when:

Long press (Menu) reliably starts/stops recording, with audible earcons on each toggle.

Recording indicator (timer and/or pulsing icon) is visible during recording across all views.

On recording stop, AR client sends audio to workflow WebSocket.

Server returns note_short, and glasses display it automatically (view forced to NOTE_SHORT).

Single press (Return) cycles views in Scheme 1 order:
NOTE_SHORT → ADVICE_SHORT → NOTE_FULL → ADVICE_FULL.

Wheel scroll scrolls within the current view and does not change modes.

When cycling to a non-cached view, client fetches it via get_latest and displays it.

Server stores turn outputs under the patient/session directory structure.
